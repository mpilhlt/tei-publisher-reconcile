# Testing a TEI Publisher v10 reconciliation profile

TEI Publisher v10 is **not** the old monolithic app. It is generated by **Jinks**, an application
manager that composes an eXist-db app from reusable **profiles**. The reconciliation **server** is
a new feature profile; the reconciliation **client** lives in the existing `annotate` profile.
This skill is the procedure for getting fast, trustworthy feedback on that profile's XQuery.

The golden rule still holds: **never trust XQuery you have only read.** eXist's type system,
on-demand compilation, roaster's request binding, and Jinks template expansion all produce
surprises that only appear at runtime. Get the code generated into a live app and assert on real
HTTP responses.

Read the references as needed — they carry the detail this body only summarises:

- `references/jinks-profiles.md` — profile anatomy, how to build the reconcile **server** profile
  (modelled on `dts`/`registers`) and add the **client** to `annotate`, the jinks-cli dev loop,
  the central **CORS origin-whitelist**, container lifecycle (podman), credentials.
- `references/cypress-testing.md` — the project's own harness (Cypress +
  `cypress-ajv-schema-validator`): where tests live, the custom commands, how to run, how to
  validate reconciliation responses against the official JSON Schemas.
- `references/reconciliation-spec.md` — endpoint map, 0.2 vs 1.0 differences, exact schema
  filenames to vendor and validate against.
- `references/testbench-automation.md` — why the **local** testbench is the default gate in v10,
  how to run it, and the optional headless-Playwright path.

## The test pyramid (cheapest layer first)

Most iterations stay in layers 1–3; only gate "done" on layer 4.

1. **Ad-hoc XQuery (no HTTP).** Run a snippet directly in eXist to check pure logic — query
   parsing, scoring, manifest assembly, `xs:*` coercion. `scripts/ad-hoc-xquery.sh`. Catches
   compile errors and type mismatches in seconds. (XQSuite `%test:*` unit tests are also available
   for reusable functions; see the Cypress reference for where they fit.)
2. **Deploy, then HTTP integration in Cypress.** This is the project's native harness and your
   main loop. `cy.api()` (which sets a localhost `Origin`, so it is CORS-clean) hits the route;
   `cy.validateSchema()` checks the body against the reconciliation JSON Schema. See
   `assets/reconcile.cy.js.example`.
3. **CORS check.** The testbench calls cross-origin, so verify the preflight and that the app
   echoes the caller's `Origin` (governed by `config:origin-whitelist`). `scripts/cors-check.sh`.
4. **Reconciliation testbench (0.2 AND 1.0), run locally.** The closest thing to an authoritative
   judge. Run it **locally** — the stock origin-whitelist already allows `localhost`, so no
   security-relevant config change is needed. Gate "done" on this. See
   `references/testbench-automation.md`.

## The dev loop (v10 / jinks-cli)

You do **not** hand-edit a deployed `custom-api.json` and you do **not** redeploy a `.xar` each
edit. You edit **profile source**, sync it into the database, and **regenerate** the consuming app.

```
edit profile source  (config.json, *-api.tpl.json, *-api.xql, test/cypress/.../*.cy.js)
        │
        ▼
jinks watch ./<profile>      ← live-syncs your local profile edits into its DB collection
        │                       (run once in the background; leave it running)
        ▼
jinks update <app>           ← re-applies profiles to the generated app (expands templates,
        │                       preserves unrelated local edits, reports conflicts)
        ▼
ad-hoc XQuery  →  Cypress API tests  →  cors-check     ← inner loop
        │
        ▼  (green and stable)
local testbench 0.2 AND 1.0  ← conformance gate
```

`scripts/up.sh` brings up the container and waits for readiness. `scripts/watch-profile.sh` and
`scripts/regenerate.sh` wrap the two jinks-cli steps. First-time creation of the app uses
`jinks create <app> -c <config>` (see the jinks-profiles reference).

If a change doesn't take effect: confirm `jinks watch` actually synced it (it prints each upload),
then that you re-ran `jinks update <app>`; remember **`.tpl.*` files must be regenerated** so the
template expands — uploading the raw `.tpl.json` does nothing on its own. **Confirm the running
code is the new code before debugging logic.**

## Bring up the platform

The host is Linux (WSL/Ubuntu or native Arch) using **podman** (drop-in for docker).

```bash
scripts/up.sh           # podman run -p 8080:8080 -v exist-data:/exist/data existdb/teipublisher:latest, then waits
```

eXist boots in ~30–90s; `up.sh` polls readiness. Jinks lives at `http://localhost:8080/exist/apps/jinks`;
generated apps at `http://localhost:8080/exist/apps/<abbrev>`. Default admin password is empty;
jinks-cli's default app user is `tei`/`simple`. Full lifecycle and credentials are in the
jinks-profiles reference.

## Where the code lives (confirm against the repo)

A feature profile that serves an API declares it in `config.json` and provides two files:

- `config.json` → `api: [{ "prefix": "recon", "id": "http://…/api/reconcile",
  "path": "reconcile-api.xql", "spec": "reconcile-api.json" }]` and `depends: ["base10", "registers"]`.
- `modules/reconcile-api.tpl.json` — the OpenAPI 3.0.3 spec (Jinks template; expands to
  `reconcile-api.json`). Routes carry `operationId: "recon:<fn>"`.
- `modules/reconcile-api.xql` — roaster handlers `declare function recon:<fn>($request as map(*))`.

The reconciliation **client** goes into the `annotate` profile's `resources/scripts/annotations/`
and is wired into its entity-authority UI; keep it **decoupled** from the server via a configurable
endpoint URL (do not hard-depend on the server profile).

## CORS in v10 (read before "fixing" it)

CORS is applied **centrally** by the base profile's controller, which echoes the request `Origin`
back as `Access-Control-Allow-Origin` **only if it matches `config:origin-whitelist`**. By default
that whitelist allows only `localhost`/`127.0.0.1`. Consequences:

- The **local** testbench (`http://localhost:3000`) and Cypress (`cy.api` uses a localhost Origin)
  work out of the box — no change needed.
- The **hosted** testbench (`https://reconciliation-api.github.io`) is **blocked** by default. Only
  widen the whitelist if you deliberately choose the hosted route — that is a security-relevant
  config change. The decided default here is **local**, so you should not need to touch it.
- Methods (`GET, POST, …, OPTIONS`) and headers (`Content-Type, …`) are already allowed centrally.
  Do **not** add per-handler CORS code; if a route fails CORS, the issue is the whitelist, not the
  handler.

## Definition of done

- [ ] Ad-hoc / XQSuite logic checks pass.
- [ ] Cypress API tests green: manifest (GET, no params) + reconcile (POST) + any sub-services.
- [ ] Those responses **validate against the reconciliation JSON Schemas** for **both** 0.2 and 1.0.
- [ ] `cors-check.sh` green for every route the testbench calls (local Origin echoed; preflight ok).
- [ ] **Local** reconciliation testbench **0.2** and **1.0** both run clean (save the result/screenshot).
- [ ] Re-run once against a freshly recreated container / `jinks update --all` to prove no reliance
      on leftover DB state.
